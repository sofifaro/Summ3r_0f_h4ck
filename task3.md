Анализ цифровых следов после инцидента в приложении для управления профилями, просмотра структуры команды, работы с проектами, получения уведомлений и восстановления доступа к аккаунту.
-------------------------------------------------------------------------------------------------------------------------------------
**Что произошло?**
1. У злоумышленника были валидные логин и пароль от учетной записи одного из сотрудников, поэтому сначала был совершен вход в аккаунт сотрудника "Alex Ivanov", id сотрудника: 1042.
2. После, злоумышленник обратился к endpoint `/api/v1/users?limit=50`, чтобы посмотреть список сотрудников, их роли и персональные данные. Пример данных, которые получил атакующий:
```
{
  "total": 6,
  "items": [
    {
      "id": 1001,
      "username": "john",
      "full_name": "John Petrov",
      "email": "john@acme.local",
      "role": "admin",
      "department": "IT"
    }, ......
```
3. Чтобы закрепиться на аккаунте жертвы, злоумышленник сменил email сотрудника на свой (`"alex.personal@mailbox.test"`). Уже после успешной смены email у сотрудника, атакующий сменил email на аккаунте администратора "John Petrov", id сотрудника: 1001. Новый emai админа `"john-temp@external.test"`.
4. Обративщись на endpoint `/api/v1/notifications?user_id=1042`, злоумышленник прочитал уведомления сотрудника "Alex Ivanov". Потом прочитал уведомления администратора. Пример данных, которые получил атакующий:
```
[
  {
    "id": 48872,
    "type": "profile_change",
    "subject": "Your email was changed",
    "preview": "Your account email has been changed to john-temp@external.test",
    "created_at": "2026-03-11T09:14:07Z"
  }
]
```
5. Благодаря функционалу восстановления доступа к аккаунту, атакующий отправил reset token на уже измененный email администратора. Пример запроса, который был отправлен:
```
POST /api/v1/auth/password/forgot HTTP/1.1
Host: portal.acme.local
Content-Type: application/json

{
  "email": "john-temp@external.test"
}
```
Пример ответа, который был получен:
```

HTTP/1.1 200 OK
Content-Type: application/json

{
  "status": "ok",
  "message": "If account exists, reset instructions have been sent"
}
```
6. Уже после, снова обративщись к endpoint `/api/v1/notifications?user_id=1001`, злоумышленник смог прочитать reset token из уведомлений сотрудника. Само уведомление:
```
 {
    "id": 48873,
    "type": "password_reset",
    "subject": "Password reset requested",
    "preview": "Reset token: 4f9d2e77-7ac0-41e2-93d7-1cb7be7dd9d4",
    "created_at": "2026-03-11T09:15:02Z"
  }
```
7. Потом последовал запрос на смену пароля от аккаунта администратора, таким образом атакующий смог закрепиться в аккаунте админа.
8. В аккаунте администратора злоумышленник, путем обращение к данному endpoint `/api/v1/admin/dashboard`, прочитал информацию о текущих проектах, пользователях и активных ссесиях.
9. Последним шагом стало экспортирование данных о проектах и просмотр логов, путем обращения к данным endpoints: `/api/v1/admin/projects/export?format=json` и `/api/v1/admin/audit?limit=5`
-------------------------------------------------------------------------------------------------------------------------------------
**Почему это стало возможным?**

1. Слабая политика безопасности:
   - Простой пароль от аккаунта сотрудника.
   - Меры предотвращения в политике безопастности были отключены:
   ```
   {
    "email_change_requires_reauth": false,
    "password_reset_invalidate_sessions": false,
    "admin_mfa_required": false,
    "notify_on_sensitive_changes": true
    }
   ```
   - Медленное реагирование департамента безопасности на подозрительные действия. (Смена email, а потом и пароля на аккаунте администратора)
2. Наличие веб-узявимостей в приложении.

Какие рекомендации по исправлению ситуации следует дать клиенту?


1. Если доступ к акканту сотрудиника был получен перебором его пароля, то в первую очередь стоит ввести сильную политику паролей. Требования к ней:
   - не менее 8–12 символов
   - использование букв верхнего и нижнего регистра, цифр и специальных символов
   - запрет на использование очевидных комбинаций (даты рождения, qwerty, password)
   - регулярная смена паролей (например, каждые 90 дней)
   - запрет на повторное использование предыдущих паролей (минимум 5 последних)
   - использование уникальных паролей для разных систем внутри компании
     
2. Добавить защиту от перебора:
   - ограничение попыток ввода: блокировка после 3-5 неудачных попыток
   - временная задержка между попытками ввода
   - реализация CAPTCHA после нескольких неудачных попыток
     
3. Если доступ к паролю сотрудника был получен путем фишинга, то провести обучение сотрудников на тему "Как распознавать фишинговые сайты или письма".
   
4. Правильно настроить политику безопасности:
   - "email_change_requires_reauth": true, если email изменяется, то сотрудник будет вынужден повторно зайти в свой аккаунт
   - "password_reset_invalidate_sessions": true, если password изменяется, то сотрудник будет вынужден повторно зайти в свой аккаунт, остальные ссессии будут невалидны
   - "admin_mfa_required": true, необходимо использовать двухфакторную аутентификацию для аккаунтов администратора (добавить контрольный вопрос при входе в аккаунт), желательно ввести такую меру для всех сотрудников, а не только для админов

5. Исправить уязвимости IDOR в следующий endpoints:
   - `/api/v1/users?limit=50`
   - `/api/v1/users/1001`
   - `/api/v1/notifications?user_id=1042`

6. Не показывать токен для сброса пароля в уведомлениях, он должен храниться в бд и отправляться на email сотрудника.

7. Настроить реагирование департамента безопасности на подобные инциденты.
