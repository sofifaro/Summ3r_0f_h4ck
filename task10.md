1. Вы получили шелл с правами пользователя user, файл `/etc/shadow` имеет следующие права: `rw----rw- root user /etc/shadow`. Можно ли повысить привилегии в таком случае? Если можно, то как? Если нельзя, опишите почему и есть ли способы обхода?

**Ответ:** Нет, тк пользователь находиться в группе самого файла, а права для группы --- (то есть запрещают доступ ко всему).

***Почему так?***

1. Сам файл имеет права `rw----rw-`, владельца root и группу `user`.
2. Наш user (чей шелл мы получили) скорее всего также относиться к группе `user`. Почему? При создании нового пользователя, Linux автоматически создает группу с таким же названием. (То есть для user скорее всего была создана группа с одноименным названием)
3. Почему у файла `/etc/shadow` группа user? Скорее всего ошибочное выполнение команды от администратора (`chown root:user /etc/shadow`). (Обычно у этого файла группа root или shadow, либо группу поменяли случайно, либо это специально в рамках задания:))
4. Чтобы сделать что-то с файлом или повысить привелегии с помощью него, нужно иметь к нему доступ хотя бы какой-то.
5. В Linux порядок проверки доступа такой: сначала смотрим является ли пользователь владельцем? Нет, значит права владельца не приминимы.
6. Входит ли пользователь в группу файла? В нашем случае скорее да, поэтому применим права группы.
7. (Уже третим шагом применятся права others, но я всё же считаю что для нашего user здесь будут права группы)
8. У группы пользователя нету никаких прав на файл (то есть - - -), поэтому ни прочитать, ни записать что-то в файл мы не можем, а соответсвенно повысить привелегии с помощью него тоже не получится.
9. Почему нельзя обойти? Первое что пришло в голову, это создать другого user, чтобы он входил в группу others и тогда уже можно что-то сделать с файлом (тк у группы others права rw-)
10. Но для создания другого user нужны следующие команды: `useradd`, `adduser`, а уже для них требуются права на запись в `/etc/passwd`, `/etc/shadow`, `/etc/group`, коих у нас нет.
11. Поэтому мы застреваем в этой группе и не можем никак обойти это ограничение, тк у нас буквально нет никаких прав.

2. Какой будет ответ, если у пользователя user будет такой набор групп: `user;cdrom;vboxusers;docker`, а разрешения у файла `/etc/shadow` будут такими? `rw----rw- root adm /etc/shadow`

**Ответ:** Да, тк пользователь находиться не в группе самого файла, а права для others rw- (то есть разрешают чтение и запись файла).

***Почему так?***

1. (Здесь не буду так подробно про группы расписывать, можно обращаться к ответу 1 для более подробного описания)
2. У нашего пользователся группы `user;cdrom;vboxusers;docker`, а группа файла `adm`.
3. Теперь нам явно дают понять, что у нашего user группы, отличные от `user`, поэтому мы можем говорить, что на файл `/etc/shadow` применятся права группы others (то есть rw-).
4. Теперь у нас есть доступ к чтению и записи в файле, так что мы можем переписать пароль у пользователя root.
5. Редактируем файл (через vi или nano), тут либо можно удалить хеш пароля: `root::...` (и пароль станет пустым), либо заменить на собственный (к примеру `mkpasswd -m sha-512`)
6. Выполняем `su root` или входим в систему без пароля, и всё, у нас есть полный конроль над системой.
